# 01 插件开发总览

## 1. 插件边界

KeySwift 插件用于扩展宿主功能，而不是替代宿主核心。

宿主核心保留：

- 用户注册、登录和用户中心基础能力。
- 后台登录、管理员和基础权限能力。
- 商品基础信息、分类和图片展示能力。
- 订单创建和订单正式状态裁决。
- 默认余额支付。
- 余额账户和账务主链。
- 插件发现、注册、启停、配置、权限、数据库声明和审计。

插件适合承载：

- 第三方支付方式。
- 发卡和交付方式。
- 商品类型扩展。
- 客服、工单、营销、优惠券等功能扩展。
- 通知、外部平台集成、对账、导入导出。
- 主题、后台工作区、页面和组件扩展。

插件不得承载：

- 正式用户主表。
- 正式管理员主表。
- 正式会话主表。
- 正式权限授权主表。
- 正式订单主表。
- 正式余额账本。
- 宿主插件治理状态。

## 2. 插件包目录

推荐目录：

```text
plugins/
└── payment-alipay/
    └── releases/
        └── 1.0.0/
            ├── manifest.json
            ├── checksums.json
            ├── bin/
            │   └── windows-amd64/
            │       └── payment-alipay.exe
            ├── frontend/
            │   └── admin.bundle.js
            ├── assets/
            ├── locales/
            └── docs/
```

规则：

- `manifest.json` 是插件声明入口。
- `checksums.json` 是完整性校验入口。
- 多版本插件按 `releases/<version>` 隔离。
- 插件不得把数据库 schema 文件直接丢给宿主执行，必须先在 manifest 中声明数据库结构。

## 3. 生命周期

插件生命周期：

1. 发现：宿主扫描插件目录。
2. 解析：读取 manifest。
3. 校验：检查协议、平台、完整性、数据库声明、权限声明。
4. 登记：写入插件治理表。
5. 审批：管理员确认启用或停用。
6. 结构处理：管理员显式执行插件数据库结构创建或校验。
7. 启动：宿主拉起插件进程或加载插件前端入口。
8. 握手：插件提交运行时能力快照。
9. 运行：插件通过受控接口与宿主交互。
10. 停用：宿主关闭流量和运行态。

## 4. 插件类型

| 类型 | pluginKind | 说明 |
| --- | --- | --- |
| 功能插件 | `functional` | 发卡、商品扩展、客服、营销等业务能力 |
| 支付插件 | `integration` 或 `functional` | 第三方支付渠道、回调、对账、退款 |
| 主题插件 | `ui-theme` | 主题 token、布局皮肤、组件覆盖、图标包 |
| 工具插件 | `tooling` | 导入导出、诊断、维护、批处理 |

## 5. 插件分类

插件分类写入 `identity.categories`，用于后台插件管理页展示、检索和动态筛选。

`pluginKind` 和 `identity.categories` 的职责不同：

- `pluginKind` 表示插件技术形态，保持少量稳定枚举。
- `identity.categories` 表示插件业务分类，可随插件生态扩展。
- 后台分类筛选项来自当前已安装插件真实声明的分类，不是宿主写死列表。
- 没有插件声明的分类不应出现在后台筛选项中。
- 未声明分类的插件仍可在“全部分类”中显示。

示例：

```json
{
  "pluginKind": "integration",
  "identity": {
    "displayName": "Cloudflare Turnstile",
    "categories": ["security", "human-verification"]
  }
}
```

## 6. 插件与宿主交互原则

- 插件声明能力，宿主登记能力。
- 插件提交事实，宿主裁决正式状态。
- 插件只访问自己的数据表。
- 插件通过事件、任务、配置和受控 API 与宿主联动。
- 插件不能直接改宿主核心状态。
